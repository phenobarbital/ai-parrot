# Feature Specification: Move PageIndex under `parrot.knowledge`

**Feature ID**: FEAT-198
**Date**: 2026-05-28
**Author**: Jesus Lara
**Status**: approved
**Target version**: 0.26.x

---

## 1. Motivation & Business Requirements

### Problem Statement

AI-Parrot currently has two knowledge-indexing subsystems with asymmetric placement in the package tree:

- **GraphIndex** — *Structured Knowledge Graph Indexing* — lives correctly at `packages/ai-parrot/src/parrot/knowledge/graphindex/`.
- **PageIndex** — *Tree-indexing (vectorless, reasoning-based RAG)* — lives at the top level `packages/ai-parrot/src/parrot/pageindex/`, instead of alongside its sibling subsystem inside `parrot.knowledge`.

Both subsystems answer the same architectural question ("how do we index and retrieve knowledge?") and even cross-call each other (`graphindex.builder` and `graphindex.extractors.loader` import from `parrot.pageindex`). Having one at top-level and the other under `parrot.knowledge` is an inconsistency that misleads new contributors and breaks the "every knowledge backend lives under `parrot.knowledge`" mental model.

### Goals

- Move `packages/ai-parrot/src/parrot/pageindex/` → `packages/ai-parrot/src/parrot/knowledge/pageindex/` so the two indexing backends sit as siblings.
- Update **every** import in the repo to the new path (`parrot.knowledge.pageindex.*`).
- **No shim imports**: do not leave a `parrot/pageindex/__init__.py` that re-exports from `parrot.knowledge.pageindex`. Importers must reference the canonical path.
- Rename the package logger from `"parrot.pageindex"` to `"parrot.knowledge.pageindex"`.
- Relocate tests from `packages/ai-parrot/tests/test_pageindex/` → `packages/ai-parrot/tests/knowledge/pageindex/` to mirror the source layout and match the existing `tests/knowledge/graphindex/` convention.
- Update documentation references (`docs/pageindex.md`, in-tree docstrings) to the new import paths.
- Leave the user-facing toolkit identity (`PageIndexToolkit.name = "pageindex"`, `tool_prefix = "pageindex"`) unchanged — those are configuration keys, not module paths.

### Non-Goals (explicitly out of scope)

- Any behavioral change inside the PageIndex pipeline (builder, retriever, ingester, toolkit). This is a pure relocation.
- Renaming or refactoring of `PageIndex*` classes or public APIs.
- Modifying `parrot.knowledge.__init__.py` to re-export PageIndex symbols (the strict layout matches the existing `parrot.knowledge.graphindex` style — graphindex symbols are not re-exported at `parrot.knowledge` either).
- Changing the `PageIndexToolkit` user-facing `name`/`tool_prefix` strings (they are config-coupled and renaming them is a breaking change beyond a code move).
- Editing **historical** SDD documents that quote the old path (completed tasks, archived brainstorms, prior specs). These are immutable history.
- Hand-editing build artifacts under `packages/ai-parrot/build/lib.*/parrot/pageindex/` — they are regenerated by the next build.

## 2. Architectural Design

### Overview

This is a **pure relocation** refactor — no logic changes. Every `.py` file in the current `parrot/pageindex/` directory moves verbatim to `parrot/knowledge/pageindex/`, with three classes of edits applied during the move:

1. **Logger names** — `logging.getLogger("parrot.pageindex")` → `logging.getLogger("parrot.knowledge.pageindex")` in every file that defines one.
2. **Docstring path references** — `parrot.pageindex.module.symbol` → `parrot.knowledge.pageindex.module.symbol` inside docstrings (e.g. `pdf_to_markdown.py`, `ingest.py`, `utils.py`).
3. **External imports** — every `from parrot.pageindex...` or `import parrot.pageindex...` (in production source, tests, examples, docs) becomes `from parrot.knowledge.pageindex...`.

Internal **relative imports** inside `pageindex/` (`from .schemas import …`, `from .hybrid_search import …`) require no edits — they remain relative to the package and will resolve correctly at the new location.

The old directory is **deleted** entirely. No `parrot/pageindex/__init__.py` shim is left behind.
