## Overview

`FieldType.REST` represents a form field whose value is resolved by an
external service (remote HTTP, internal loopback, or a registered Python
callback). The frontend uploads a file via multipart POST and receives a
structured JSON envelope.

---

## JSON Schema Extension: `x-parrot-rest`

When the backend renders a form to JSON Schema (e.g. for an OpenAPI spec),
REST fields carry the `x-parrot-rest` extension key:

```json
{
  "type": "string",
  "title": "Planogram Photo",
  "x-parrot-rest": {
    "mode": "callback",
    "callback_ref": "planogram_analyzer",
    "response_path": "$.result.label",
    "display_template": "Detected: {{ value }}"
  }
}
```

The extension shape mirrors the `RestFieldSpec` model (see §Spec Models below).

## Upload Endpoint

```
POST /api/v1/forms/{form_id}/fields/{field_id}/upload
Content-Type: multipart/form-data
Authorization: Bearer <token>
X-Parrot-Prior-Blob-Ref: <previous blob URI>  (optional — triggers deletion)
```

### Request

| Part | Required | Notes |
|---|---|---|
| `file` (binary part) | yes | The file to upload. Content-Type set per part. |

**`X-Parrot-Prior-Blob-Ref`** — if provided, the backend deletes the previous
blob after the new one is durably stored. Deletion failure does NOT cause a
non-200 response; it appears as a string in `warnings`.

### Error Codes

| Code | Meaning |
|---|---|
| 400 | Malformed multipart, missing file part, or invalid REST spec |
| 401 | Unauthenticated |
| 404 | Form not found or field is not `FieldType.REST` |
| 413 | File exceeds `constraints.max_file_size_bytes` |
| 415 | MIME type not in `constraints.allowed_mime_types` |
| 500 | Server misconfiguration (blob_storage or rest_resolver not wired) |

### Response Envelope (HTTP 200)

A `success: false` result from the resolver is returned as HTTP **200**
(not 500). The frontend must inspect `success` to determine outcome.

```json
{
  "success": true,
  "answer": "<JSONPath-extracted value or raw_value>",
  "raw_value": "<full resolver response>",
  "blob_ref": "s3://bucket/form_id/field_id/<uuid>",
  "display": "<Jinja2-rendered display string or null>",
  "warnings": [],
  "error": null
}
```

## YAML Form Definition Example — Planogram Analyzer

```yaml
field_id: planogram_image
field_type: rest
label: Planogram Photo
constraints:
  allowed_mime_types:
    - image/jpeg
    - image/png
  max_file_size_bytes: 10485760   # 10 MB
meta:
  rest:
    mode: callback
    callback_ref: planogram_analyzer
    response_path: "$.result.label"
    display_template: "Detected: {{ value }}"
```

The planogram analyzer callback (`planogram_analyzer`) receives the image
bytes, runs detection, and returns a `RestCallbackOutput` with
`value={"result": {"label": "Facing OK"}}`. The `response_path`
`$.result.label` extracts `"Facing OK"` into `answer`, and
`display_template` renders it as `"Detected: Facing OK"`.

## Integration Checklist

- [ ] Render `FieldType.REST` fields as a file-upload input.
- [ ] On upload, `POST` multipart to the upload endpoint.
- [ ] Attach `Authorization: Bearer <token>` (session token).
- [ ] Attach `X-Parrot-Prior-Blob-Ref` if re-uploading to the same field.
- [ ] Inspect `success` in the response envelope (200 ≠ success).
- [ ] Display `display` string if present, otherwise fall back to `answer`.
- [ ] Surface `warnings[]` to the user if non-empty.
- [ ] Show `error` message when `success: false`.
