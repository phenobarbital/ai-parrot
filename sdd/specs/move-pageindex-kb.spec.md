# Feature Specification: Move PageIndex under `parrot.knowledge`

**Feature ID**: FEAT-198
**Date**: 2026-05-28
**Author**: Jesus Lara
**Status**: approved
**Target version**: 0.26.x

---

## 1. Motivation & Business Requirements

### Problem Statement

AI-Parrot currently has two knowledge-indexing subsystems with asymmetric placement in the package tree:

- **GraphIndex** ‚Äî *Structured Knowledge Graph Indexing* ‚Äî lives correctly at `packages/ai-parrot/src/parrot/knowledge/graphindex/`.
- **PageIndex** ‚Äî *Tree-indexing (vectorless, reasoning-based RAG)* ‚Äî lives at the top level `packages/ai-parrot/src/parrot/pageindex/`, instead of alongside its sibling subsystem inside `parrot.knowledge`.

Both subsystems answer the same architectural question ("how do we index and retrieve knowledge?") and even cross-call each other (`graphindex.builder` and `graphindex.extractors.loader` import from `parrot.pageindex`). Having one at top-level and the other under `parrot.knowledge` is an inconsistency that misleads new contributors and breaks the "every knowledge backend lives under `parrot.knowledge`" mental model.

### Goals

- Move `packages/ai-parrot/src/parrot/pageindex/` ‚Üí `packages/ai-parrot/src/parrot/knowledge/pageindex/` so the two indexing backends sit as siblings.
- Update **every** import in the repo to the new path (`parrot.knowledge.pageindex.*`).
- **No shim imports**: do not leave a `parrot/pageindex/__init__.py` that re-exports from `parrot.knowledge.pageindex`. Importers must reference the canonical path.
- Rename the package logger from `"parrot.pageindex"` to `"parrot.knowledge.pageindex"`.
- Relocate tests from `packages/ai-parrot/tests/test_pageindex/` ‚Üí `packages/ai-parrot/tests/knowledge/pageindex/` to mirror the source layout and match the existing `tests/knowledge/graphindex/` convention.
- Update documentation references (`docs/pageindex.md`, in-tree docstrings) to the new import paths.
- Leave the user-facing toolkit identity (`PageIndexToolkit.name = "pageindex"`, `tool_prefix = "pageindex"`) unchanged ‚Äî those are configuration keys, not module paths.

### Non-Goals (explicitly out of scope)

- Any behavioral change inside the PageIndex pipeline (builder, retriever, ingester, toolkit). This is a pure relocation.
- Renaming or refactoring of `PageIndex*` classes or public APIs.
- Modifying `parrot.knowledge.__init__.py` to re-export PageIndex symbols (the strict layout matches the existing `parrot.knowledge.graphindex` style ‚Äî graphindex symbols are not re-exported at `parrot.knowledge` either).
- Changing the `PageIndexToolkit` user-facing `name`/`tool_prefix` strings (they are config-coupled and renaming them is a breaking change beyond a code move).
- Editing **historical** SDD documents that quote the old path (completed tasks, archived brainstorms, prior specs). These are immutable history.
- Hand-editing build artifacts under `packages/ai-parrot/build/lib.*/parrot/pageindex/` ‚Äî they are regenerated by the next build.

## 2. Architectural Design

### Overview

This is a **pure relocation** refactor ‚Äî no logic changes. Every `.py` file in the current `parrot/pageindex/` directory moves verbatim to `parrot/knowledge/pageindex/`, with three classes of edits applied during the move:

1. **Logger names** ‚Äî `logging.getLogger("parrot.pageindex")` ‚Üí `logging.getLogger("parrot.knowledge.pageindex")` in every file that defines one.
2. **Docstring path references** ‚Äî `parrot.pageindex.module.symbol` ‚Üí `parrot.knowledge.pageindex.module.symbol` inside docstrings (e.g. `pdf_to_markdown.py`, `ingest.py`, `utils.py`).
3. **External imports** ‚Äî every `from parrot.pageindex...` or `import parrot.pageindex...` (in production source, tests, examples, docs) becomes `from parrot.knowledge.pageindex...`.

Internal **relative imports** inside `pageindex/` (`from .schemas import ‚Ä¶`, `from .hybrid_search import ‚Ä¶`) require no edits ‚Äî they remain relative to the package and will resolve correctly at the new location.

The old directory is **deleted** entirely. No `parrot/pageindex/__init__.py` shim is left behind.

### Known Risks / Gotchas

- **Monkeypatch target strings are not caught by static import analysis.** The 65 `parrot.pageindex` occurrences in `tests/test_pageindex/` include `monkeypatch.setattr("parrot.pageindex.utils.count_tokens", ‚Ä¶)` patterns. These resolve via string lookup at test time and will silently break if missed. The repo-wide grep in ¬ß5 Module 5 is the safety net.
- **Lazy imports inside functions.** `graphindex/extractors/loader.py:351` does `from parrot.pageindex import md_to_tree` inside a method body, not at module top. Pure grep catches it; static import resolvers may not. Don't rely on `ruff`'s import inspection alone.
- **`docs/pageindex.md` is user-facing.** If this page is published (mkdocs), an out-of-date snippet creates a copy-paste failure for users. Update every code block, not just imports.
- **Historical SDD documents** (`sdd/tasks/completed/`, `sdd/proposals/`, prior `sdd/specs/`) reference the old path and **must remain unchanged**. They document history. The grep in ¬ß5 explicitly excludes them.
- **Editable install (`pip install -e .`)**: contributors with an existing `.venv` may have the **old** `parrot/pageindex` cached in `__pycache__`. Document in the PR description that contributors should run `pip install -e packages/ai-parrot/ --force-reinstall` after pulling, or simply delete `__pycache__/` directories.
- **Log filtering downstream.** Any external log shipper / dashboard / alert filtering on logger name `parrot.pageindex` will silently stop matching. If such filters exist in `services/` or deployment configs, update them. This spec does NOT enumerate operational configs ‚Äî flag any found during implementation as a follow-up.
- **No examples/pageindex_compliance_agent renaming** even though `examples/pageindex/` is a directory ‚Äî directory paths under `examples/` are not Python module paths. Leave the directory layout alone; only edit imports inside the files.
