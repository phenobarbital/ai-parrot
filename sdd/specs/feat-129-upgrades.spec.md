# Feature Specification: dev-loop intake upgrades — intent routing + plan-summary comment

**Feature ID**: FEAT-132
**Date**: 2026-04-28
**Author**: Jesus Lara
**Status**: approved
**Target version**: 0.6.x (`ai-parrot`, post-FEAT-129)

---

## 1. Motivation & Business Requirements

### Problem Statement

The dev-loop flow shipped by FEAT-129 (`sdd/specs/dev-loop-orchestration.spec.md`)
treats every incoming `BugBrief` as a *bug*: it always opens a Jira ticket of
type `Bug` and unconditionally routes through `BugIntakeNode` before research.
In practice the same orchestrator is also being used to drive *enhancements*
and *new features*, where:

- The Jira ticket type must NOT be `Bug` (those are work items, not defects).
- The "intake" step that validates a stack-trace shape and grabs CloudWatch
  excerpts is largely irrelevant — there is no incident to triage.
- The agent should make explicit, BEFORE doing anything heavy, what plan it
  intends to follow. Today the planning is implicit inside the
  `sdd-research` dispatch and never lands as a Jira comment, so the human
  reviewer can't sign off on the approach without spelunking the dispatch
  log.

This upgrade adds a tiny intake-time decision step ("is this a bug or a work
item?") and a plan-summary that lands on the Jira ticket as the first
comment when the ticket is freshly created.

### Goals

- **G1** — `WorkBrief.kind ∈ {"bug", "enhancement", "new_feature"}` is
  declared by the caller (UI radio button or programmatic API). The model
  is renamed from `BugBrief` to `WorkBrief`; a `BugBrief = WorkBrief`
  alias preserves all existing imports.
- **G2** — A new `IntentClassifierNode` is inserted at the head of the
  flow. It validates the brief and **routes** by `kind`:
  - `kind == "bug"` → `BugIntakeNode` → `ResearchNode`
  - `kind ∈ {"enhancement", "new_feature"}` → `ResearchNode` directly,
    skipping `BugIntakeNode`.
- **G3** — `ResearchNode.jira_create_issue` picks the Jira issuetype from
  a fixed `kind → issuetype` map: `bug → "Bug"`, `enhancement → "Story"`,
  `new_feature → "New Feature"` (the user's NAV instance accepts those
  three labels — verified in conversation 2026-04-28).
- **G4** — When the ticket is **newly created** (i.e. neither
  `WorkBrief.existing_issue_key` nor the JQL summary search yielded a
  match), `ResearchNode` posts the LLM-generated plan summary as the
  ticket's first comment, headed by `Plan for run-<run_id>`. On reuse
  paths (re-trigger), the existing
  `Dev-loop re-triggered for this ticket (run_id=…)` comment stays the
  only post — no plan-summary on reuse.
- **G5** — The plan summary is generated by the existing `DEV_LOOP_SUMMARY_LLM`
  client (Haiku 4.5 by default) with a dedicated system prompt: it
  consumes `WorkBrief.summary`, `WorkBrief.description` and the freshly
  collected `log_excerpts`, and emits a short, action-oriented plan
  document. No new env knob is added unless the user wants a separate
  model — in which case `DEV_LOOP_PLAN_LLM` overrides the summarizer
  default.

### Non-Goals (explicitly out of scope)

- Re-architecting `BugIntakeNode` for non-bug kinds. It stays bug-only.
  Enhancements/new features bypass it entirely.
- LLM-driven intent classification. The user explicitly chose UI-driven
  classification (`Q1 = (a)`).
- Posting plan summaries on re-triggered tickets. The user explicitly
  chose "only on first creation" (`Q2`).
- Renaming `dev-loop-orchestration.spec.md` or any FEAT-129 contract
  beyond `BugBrief → WorkBrief` and the `kind` field. Pydantic alias +
  module-level export keep all callers working.

## 2. Architectural Design

### Overview

Insert one new node (`IntentClassifierNode`) at the start of the flow. It
absorbs `BugIntakeNode`'s universal validation responsibilities (allowlist
heads, path-traversal checks, brief shape), emits a single
`flow.intake_validated` event with `kind` to the per-run flow stream, then
routes by `kind`. `BugIntakeNode` is preserved as a bug-only enrichment
hook (currently a near-no-op besides emitting `flow.bug_brief_validated`)
so future work can attach severity classification, stack-trace parsing,
etc., without re-shaping the flow.

`ResearchNode` gains a new method `_build_plan_summary(brief, excerpts)`
that runs the summarizer LLM with a plan-oriented system prompt, plus a
`_post_plan_summary_comment(issue_key, plan, run_id)` helper that writes
the comment via `JiraToolkit.jira_add_comment`. Both fire ONLY on the
new-ticket path; the reuse path (`existing_key is not None`) continues
to use the unchanged `_comment_retriggered` helper added in 8d151ee4.

### Component Diagram

```
                     ┌───────────────────────────┐
                     │ IntentClassifierNode      │

                     │  routes by kind           │
```

### Data Models

```python
# parrot/flows/dev_loop/models.py
from typing import Literal

WorkKind = Literal["bug", "enhancement", "new_feature"]


class WorkBrief(BaseModel):
    """Renamed from BugBrief. Carries all dev-loop intake state.

    A module-level alias `BugBrief = WorkBrief` is exported for back-
    compat. Existing code that does `from parrot.flows.dev_loop import
    BugBrief` keeps working unchanged.
    """

    kind: WorkKind = Field(
        default="bug",
        description=(
            "Intake classification: 'bug' for defect triage, "
            "'enhancement' for changes to existing behaviour, "
            "'new_feature' for net-new capability. The IntentClassifier "
            "node uses this to pick the Jira issuetype and route the "
            "flow (bugs go through BugIntakeNode; the other two skip it)."
        ),
    )
    summary: str = Field(..., min_length=10, max_length=255, ...)
    description: str = Field(default="", ...)
    affected_component: str
    log_sources: List[LogSource] = Field(default_factory=list)
    acceptance_criteria: List[AcceptanceCriterion] = Field(..., min_length=1)
    escalation_assignee: str
    reporter: str
    existing_issue_key: Optional[str] = None


BugBrief = WorkBrief  # back-compat alias
```

`CriterionResult.kind`, `LogSource.kind`, `WorkBrief.kind` are independent
literals — naming collision is intentional to keep readability inside
each model; they are unrelated runtime types.

## 3. Module Breakdown

### Module 1: `parrot.flows.dev_loop.models` — `WorkBrief` rename + `kind`
- **Path**: `packages/ai-parrot/src/parrot/flows/dev_loop/models.py`
- **Responsibility**: Rename `BugBrief` → `WorkBrief`; add `kind` field with
  literal `bug | enhancement | new_feature` (default `bug`); export alias
  `BugBrief = WorkBrief`. No other field changes.
- **Depends on**: nothing internal.

### Module 2: `IntentClassifierNode`
- **Path**: `packages/ai-parrot/src/parrot/flows/dev_loop/nodes/intent_classifier.py` (new)
- **Responsibility**: Implements the validation extracted from
  `BugIntakeNode._validate` (allowlist heads, path-traversal); emits the
  `flow.intake_validated` event to Redis; populates `ctx['bug_brief']`
  (kept for back-compat) and `ctx['work_brief']` (new); returns the
  validated `WorkBrief` so flow predicates can read `result.kind`.
- **Depends on**: Module 1.

### Module 3: `BugIntakeNode` scope-down
- **Path**: `packages/ai-parrot/src/parrot/flows/dev_loop/nodes/bug_intake.py`
- **Responsibility**: Drop the universal validation (now in Module 2).
  Keep the `flow.bug_brief_validated` Redis event so existing observers
  don't break. Becomes a thin pass-through hook reserved for future
  bug-specific enrichment (severity classification, stack-trace parsing).
- **Depends on**: Module 1, Module 2 (logical, not import).

### Module 4: `ResearchNode` upgrades
- **Path**: `packages/ai-parrot/src/parrot/flows/dev_loop/nodes/research.py`
- **Responsibility**:
  1. Add `_ISSUE_TYPE_BY_KIND` and route `issuetype` from `brief.kind`.
  2. Add `_build_plan_summary(brief, excerpts) -> str`. Uses the existing
     LLM client unless `DEV_LOOP_PLAN_LLM` is set. System prompt is
     plan-oriented ("describe the steps you would take to resolve this
     ticket; do not list logs verbatim").
  3. Add `_post_plan_summary_comment(issue_key, plan, run_id)`. Same
     call site as `_comment_retriggered`.
  4. In `execute()`, after a *new* ticket is created (the
     `existing_key is None` branch), call the new helpers in order:
     `_build_plan_summary` → `_post_plan_summary_comment`. On reuse
     path, leave the existing `_comment_retriggered` call unchanged.
- **Depends on**: Module 1, existing `LLMFactory`, `JiraToolkit`.

### Module 5: Flow factory rewire
- **Path**: `packages/ai-parrot/src/parrot/flows/dev_loop/flow.py`
- **Responsibility**: Add `IntentClassifierNode` to the adapter set.
  Replace the `task_flow(bug_intake.name, research.name)` linear edge with
  a branching topology:
  - `task_flow(intent_classifier.name, …)` → not used directly; use
    `on_condition` instead (the existing pattern for the QA pass/fail
    branch is the model).
  - `flow.on_condition(intent_classifier.name, bug_intake.name, predicate=lambda r: r.kind == "bug")`
  - `flow.on_condition(intent_classifier.name, research.name, predicate=lambda r: r.kind != "bug")`
  - `task_flow(bug_intake.name, research.name)` retained for the bug
    path.
- **Depends on**: Modules 1–4.

### Module 6: Optional plan LLM setting
- **Path**: `packages/ai-parrot/src/parrot/conf.py`
- **Responsibility**: Optional `DEV_LOOP_PLAN_LLM` setting (str, default
  `""` → falls back to `DEV_LOOP_SUMMARY_LLM`). Documented in the
  examples README.
- **Depends on**: nothing.

### Module 7: UI form + server form-builder
- **Path**: `examples/dev_loop/static/index.html`,
  `examples/dev_loop/server.py`
- **Responsibility**:
  - HTML: add a 3-radio group at the top of the form (`Bug` / `Enhancement`
    / `New Feature`), default `Bug`. Submit serialises `kind` into the
    POST body.
  - `server.py::_build_brief_from_form`: read `form.get("kind")`,
    normalise to lowercase + underscore, pass into the `WorkBrief`
    payload as `kind`. Any unknown value falls back to `"bug"` with a
    warning log.
- **Depends on**: Modules 1, 5.

### Module 8: Tests
- **Path**: `packages/ai-parrot/tests/flows/dev_loop/`
- **Responsibility**:
  - `test_intent_classifier.py` (new) — validation, routing predicate
    output, redis event emission.
  - `test_research.py` extended — new tests for issuetype routing per
    `kind`, plan-summary on first create, no plan-summary on reuse.
  - `test_flow.py` extended — assert the branching topology routes
    `bug` to BugIntakeNode and `enhancement`/`new_feature` straight to
    ResearchNode. Pre-existing flow tests stay green.
  - `test_models.py` extended — `BugBrief` alias still works,
    `WorkBrief.kind` defaults to `"bug"`, literal validation rejects
    invalid kinds.

## 7. Implementation Notes & Constraints

### Patterns to Follow

- **Async-first**: every node's `execute` is `async def`. The plan-summary
  call to `LLMFactory.create(...).ask(...)` is async.
- **Pydantic v2**: `kind` is a `Literal[...]` discriminator-friendly type.
  Use `populate_by_name=True` on `WorkBrief` only if back-compat aliases
  are added later — for v1 of this spec, the new `kind` field has no
  alias.
- **Lazy SDK access**: do not import `claude_agent_sdk` at module top.
  The plan summary uses `LLMFactory.create("claude-haiku-…")` like the
  existing summarizer; same pattern, same caveats.
- **Redis publish**: re-use the lazy `aioredis.from_url(...)` pattern
  from `bug_intake.py:_ensure_redis`. The new node must emit at most
  ONE event per dispatch.
- **Logger over print**: `self.logger.info(...)` for routing decisions,
  `warning` for fallbacks (LLM failure, unknown kind).
- **navconfig**: use `conf.config.get("DEV_LOOP_PLAN_LLM", fallback="")`
  with `fallback=` (NOT `default=` — Kardex API quirk; see
  `feedback_navconfig_kardex_fallback.md`).
- **Re-trigger comment unchanged**: do NOT touch `_comment_retriggered`.
  Only the new-ticket branch gets the new `_post_plan_summary_comment`
  call.
- **`shared_ctx` propagation**: the `_NodeAgentAdapter` in flow.py
  (commit cb244747) shares one ctx dict across nodes. The new
  `IntentClassifierNode` writes `ctx['bug_brief']` (legacy key) AND
  `ctx['work_brief']` (new key) so future migrations can pivot without
  breaking existing readers.

### Known Risks / Gotchas

- **R1 — Migrating `BugIntakeNode._validate` callers**. Tests in
  `tests/flows/dev_loop/test_bug_intake.py` exercise `_validate` via
  `BugIntakeNode.execute`. After scope-down, those tests must be
  rewritten to drive `IntentClassifierNode.execute`. Mitigation:
  literal copy of the validation logic (no semantic change), then
  `git mv`-equivalent test relocation.
- **R2 — `kind` field ordering**. Pydantic JSON schema field order
  matters for the dispatcher's `_build_prompt` (the prompt now embeds
  the schema). Putting `kind` first in `WorkBrief` keeps it visible in
  the rendered field list. Document this in the model docstring so a
  later refactor doesn't accidentally reorder.
- **R3 — Plan-summary cost**. Every newly-created ticket triggers ONE
  Haiku call (a few hundred tokens of output). Reuse path adds zero
  cost. Mitigation: reuse path is the common case once the project
  is established, so steady-state cost is small.
- **R4 — Plan-summary LLM failure**. If the model errors out, the
  flow MUST proceed (the ticket still got created; the human can
  read the description). The fallback emits a deterministic
  "Plan unavailable: <error>" body so the comment slot is still
  populated.
- **R5 — Unknown `kind` from form**. The UI radio is constrained, but
  someone hitting the API with `kind="story"` should not crash the
  form-builder. Default to `"bug"` + warning log.
- **R6 — `BugIntakeNode` becoming a near-no-op**. After scope-down it
  only emits the redis event. Tempting to remove. Decision: KEEP it as
  an empty extension hook so future work can add bug-specific logic
  without re-shaping the topology. This is documented in §1 Goals G2.
- **R7 — `on_condition` ordering**. `AgentsFlow.on_condition` evaluates
  predicates in registration order. Register the `kind == "bug"`
  predicate first so it short-circuits cleanly. Verify with a unit
  test that registers both then triggers each kind.

## Worktree Strategy

- **Default isolation unit**: **per-spec** (sequential tasks).
- All eight modules touch tightly-coupled files inside
  `parrot/flows/dev_loop/` plus the example client. Splitting them
  across worktrees would create perpetual rebase pain on
  `dev_loop/__init__.py`, `models.py`, `flow.py`, and the test fixtures.
  One feature branch, one worktree, sequential tasks.
- **Cross-feature dependencies**: hard dependency on **FEAT-129** being
  merged into `dev`. Verified: present at commits up to `cb244747`.
  No new external feature dependency.
- **Recommended worktree creation** (after `/sdd-task`):
  ```bash
  git checkout dev && git pull origin dev
  git worktree add -b feat-132-feat-129-upgrades \
    .claude/worktrees/feat-132-feat-129-upgrades HEAD
  ```
